# Launcher Optimization / Оптимизация лаунчера

Оптимизации времени запуска Minecraft в TuffBox, выполненные на основе
аудита кода и сравнения с референсными лаунчерами
(`AstralRinth-AR` — форк Modrinth/Prism, `nitrolaunch` — Rust-лаунчер).

Дата: 2026-07-09.

## Узкие места (до оптимизации)

Каждый вызов `TestLauncher::build_command` (через `launch_profile` →
`build_and_spawn` в `lib.rs`) выполнял:

1. `install_game()` — пересобирал метаданные установленной версии:
   перечитывал `version.json`, перебирал **все** библиотеки, читал asset
   index, **заново распаковывал natives** для каждого native-архива и
   заново мержил loader-профиль. Это сотни filesystem-операций на запуск.
2. `extract_natives()` — распаковывал native-библиотеки **каждый раз**,
   даже если папка natives уже заполнена.
3. `canonicalize()` для каждой библиотеки в `classpath_string` — десятки–
   сотни реальных syscall (`fs::canonicalize`) на построение classpath.

При повторных test-запусках (типичный сценарий разработки модпака) всё это
делалось заново, хотя на диске ничего не менялось.

## Применённые оптимизации

### 1. In-process кеш `InstalledVersion` (`mc_install.rs`)
- `lazy_static` `INSTALL_CACHE: Mutex<HashMap<String, InstalledVersion>>`,
  ключ — `(mc_version, loader_kind, loader_version)`.
- `install_game()` сначала проверяет кеш; если есть и **все критичные файлы
  присутствуют** (`client_jar`, все `libraries`, `natives_dir`) — возвращает
  закешированный результат без пересборки/распаковки/мержа.
- После реальной установки результат сохраняется в кеш (`store_install`).
- Валидация файлов гарантирует: удалённый/повреждённый инсталл не используется
  из кеша, а переустанавливается.

### 2. Пропуск `extract_natives` (`mc_install.rs`)
- `extract_natives()` теперь возвращается сразу, если `natives_dir` уже
  содержит хотя бы один файл. Распаковка (read+write десятков файлов)
  происходит только при первой установке.

### 3. Кеш `canonicalize` (`launcher.rs`)
- `lazy_static` `CANON_CACHE: Mutex<HashMap<PathBuf, PathBuf>>`.
- `canonicalize()` возвращает закешированный нормализованный путь вместо
  повторного `fs::canonicalize` syscall для тех же путей (libraries, dirs,
  jar). Убирает сотни syscall при построении classpath на запуск.

### 4. Дисковый кеш `InstalledVersion` (`mc_install.rs`)
- Поверх in-process кеша добавлено зеркало на диск:
  `<launcher_dir>/install-cache/<key>.json` (ключ очищен от спецсимволов).
- При запуске сначала проверяется in-process кеш, затем дисковый; оба
  валидируются по наличию файлов. Установка сохраняется в оба.
- Выгода: после перезапуска Tauri повторный launch той же версии не
  пересобирает метаданные (кеш пережил restart процесса).

### 5. Инкрементальное копирование модов/конфига (`launcher.rs`)
- `prepare_instance` теперь копирует моды и overrides только если целевой
  файл отсутствует или отличается по размеру (`is_same_file`,
  `copy_dir_incremental`). Ранее `fs::copy` перезаписывал всё при каждом
  вызове.
- Выгода: повторный launch в выделенную instance-папку (Prism-стиль) не
  переливает сотни МБ модов/конфигов — синхронизируются только изменившиеся
  файлы. Покрыто тестом `incremental_copy_skips_identical_files`.
- Примечание: основной путь запуска (`build_and_spawn`) работает прямо в
  папке проекта, где моды уже лежат, поэтому `prepare_instance` сейчас не на
  hot path (помечен `#[allow(dead_code)]` как готовая утилита для
  per-profile изоляции), но оптимизирован и протестирован.

## Что НЕ менялось

- Логика реальной установки/скачивания при первом запуске — полностью
  сохранена (сеть, параллельная загрузка библиотек через `rayon`).
- `ensure_project_mods_downloaded` — проверка наличия мод-файлов перед
  запуском оставлена (для первого запуска после добавления мода в оффлайне).
- Форматы командной строки JVM/game-аргументов — без изменений.

## Влияние

Повторный запуск (2-й, 3-й, … test run одной версии) теперь:
- не перечитывает `version.json` и не сканирует библиотеки;
- не распаковывает natives;
- не делает сотни `canonicalize` syscall;
- при использовании выделенной instance-папки не перекачивает моды/конфиг.

Экономия — порядка сотен–тысяч filesystem-операций на запуск; время до
появления окна игры сокращается (особенно заметно на HDD и при большом
числе модов). Дисковый кеш сохраняет выигрыш даже после перезапуска Tauri.

## Проверки

- `cargo check` (tuffbox-core + tauri) — без ошибок и warnings.
- `cargo test --lib` — 59 passed, 0 failed (вкл. `incremental_copy_skips_identical_files`).
- Референсные лаунчеры использованы для подтверждения паттерна
  «готовая установка переиспользуется, natives кешируются»
  (Prism-подобное поведение), однако их код (C++/web для AstralRinth,
  declarative `pkg` для NitroLaunch) не копировался напрямую.
